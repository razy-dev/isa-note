1-2. 소프트웨어 개발생명주기(SDLC) (p.005)

소프트웨어 개발생명주기(SDLC) 구성요소

타당성 조사 → 요구분석 → 설계 → 개발/구현 → 테스트 → 유지보수 → 폐기

소프트웨어 생명주기 기능

  • 프로젝트 비용 산정

  • 프로젝트 개발 계획 수립

  • 용어의 표준화

  • 개발 진행 상황의 명확한 파악

  • 단계별 문서화를 통한 프로젝트 관리

소프트웨어 생명주기 모델의 선정 기준

  • 프로젝트의 규모와 성격

  • 개발에 사용되는 방법과 도구

  • 개발에 소요되는 시간 과 비용

  • 개발 과정에서의 통제수단과 소프트웨어 산출물 인도 방식

소프트웨어 생명주기(SDLC) 모델

SDLC Model 장점 단점

폭포수 모델 (Waterfall Model) p.006

  • 가장 오래 되고 폭 넓게 사용

  • 전체 과정을 이해하기 용이

  • 관리 용이(진행 과정 세분화)

  • 초기에 요구사항 정의가 어려움

  • 중요 문제점의 발견이 늦어짐

  • 전 단계가 종결 되어야 다음 단계를 수행

프로토타입 모델 (Prototype Model) p.007

  • 요구 사항 도출이 용이

  • 개발자와 사용자간 의사소통 원활

  • 개발 타당성 검증 용이

  • 프로토타입 산출물 폐기에 따른 비경제적

  • 전체 SW 품질과 장기적 유지 보수 어려움

나선형 모델 (Spiral Model) p.009

  • 정확한 사용자 요구 사항 파악

  • 위험 부담 감소

  • 품질 확보

  • 많은 시간 소요

  • 관리에 어려움(복잡함)

  • 충분한 사례 부족 → 검증 어려움

반복적(증분/진화적) 개발 모델 p.010

RAD 모델 (Rapid Application Development) p.012

  • 요구사항의 완전한 이해와 프로젝트 범위가 명확 할 경우

  • 신속한 개발 및 고품질 구현 가능

  • 적절한 모듈화 가능성 전제

  • 기술적 위험이 높을 경우 부적합

  • 구성원의 책임감 필요

XP(eXtreme Programming)

의사소통을 개선하여 즉각적인 피드백에 의해 단순한게 코딩하여 빠르게 개발하기 위한 방법론

  • 비즈니스 상의 요구가 시시각각 변동이 심한 경우에 적합

  • 비교적 적은 규모의 인원의 개발 프로젝트에 적용

  • 개발 문서 보다는 소스코드

  • 개인의 책임과 용기에 중점

Scrum

작은 목표를 짧은 주기로 점진적이며 경험적으로, 제품을 지속적으로 개발(전달)하는 관리 프레임워크

Scrum 주요 용어
사용자 스토리(User Story)

사용자가 사용하는 관점에서 어떤 가치를 제공할 것인지를 설명

제품 백로그(Product Backlog)

개발할 제품의 요구사항인 사용자 스토리 집합이며, 우선순위로 관리

스프린트 백로그(Sprint Backlog)

각각의 스프린트 목표에 도달하기 위해 필요한 작업 목록

스프린트 계획 회의(Sprint Planning Meeting)

스프린트 목표와 스프린트 백로그를 계획

스프린트(Sprint)
번 다운 차트(burn down chart)

생산성 관리. 남아있는 일 대비 시간 그래프

칸반 보드(Kanban Board)

작업을 시각적으로 업무 상태, 흐름을 보여주는 게시판

일일 스크럼 회의(Daily Scrum Meeting)
스프린트 리뷰(Sprint Review)

스프린트 마지막날 개발자가 개발한 내용을 Stakeholder, 고객, 제품 책임자에게 시연하고 검토

스프린트 회고(Sprint Retrospective)

스프린트 마지막날 좋았던 점, 개선할 점을 도출하고 더 나은 방향으로 개선

완료 기준(Definition of Done)

Given, When, Then

인수 기준(Acceptance Criteria)

Given, When, Then

잠재적 출시 가능 제품(Potentially Shippable Product Increment)

최소 실행 가능 제품(Minimum Viable Product, MVP)

Scrum 주요 역할
스크럼 마스터(Scrum Master)

  • 팀을 보호하고 장애 요소를 해결

  • 일일 스크럼 회의를 진행

  • 모니터링 및 Tracking

제품 책임자(Product Owner)

비즈니스 목표를 충족시키는 제품을 만들기 위해 제품 백 로그를 관리하고 제품을 검토합니다.

  • 제품 백로그(요구사항) 관리/설명

  • 제품 백로그의 우선순위 관리

  • 만족스럽게 개발되었는지 확인

개발 팀 (Developer Team)

  • 최선의 기술로 백로그를 개발하여 고객을 만족시킴
